# Feature Specification: 플랫폼 산업 자동 모니터링 파이프라인

**Feature Branch**: `001-platform-monitor-pipeline`
**Created**: 2026-03-31
**Status**: Draft
**Input**: KISDI 디지털플랫폼 정책포럼을 위한 뉴스·보도자료 자동 수집 및 LLM 기반 분석 시스템

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 기관 보도자료 자동 수집 (Priority: P1)

KISDI 연구원이 시스템을 실행하면, 7대 정책영역별 대표 기관(공정거래위원회, 식품의약품안전처, 개인정보보호위원회, 고용노동부, 문화체육관광부, 행정안전부, 과학기술정보통신부)의 보도자료가 정책브리핑(korea.kr) RSS 피드를 통해 자동으로 수집된다. 수집된 보도자료는 기관명, 제목, 본문 텍스트, 원문 링크, 게시일 정보를 포함하며, 첨부파일(hwpx 등)이 있는 경우 해당 텍스트도 추출하여 저장한다.

**Why this priority**: 분석의 기초가 되는 데이터 수집이 먼저 확보되어야 후속 분석이 가능하다.

**Independent Test**: 시스템 실행 후 7개 기관의 보도자료가 JSON 파일로 저장되고, 각 항목에 제목·본문·링크·날짜가 포함되어 있는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 시스템이 인터넷에 연결된 상태, **When** 보도자료 수집 명령을 실행하면, **Then** 7개 기관별 RSS 피드에서 최신 보도자료가 수집되어 JSON 파일로 저장된다
2. **Given** 수집된 보도자료에 hwpx 첨부파일이 있는 경우, **When** 수집 처리가 완료되면, **Then** 첨부파일의 텍스트 내용이 해당 항목의 본문에 포함된다
3. **Given** 이전에 수집한 데이터가 존재하는 경우, **When** 재수집을 실행하면, **Then** 이미 수집된 항목은 중복 저장되지 않는다

---

### User Story 2 - 뉴스 기사 자동 수집 (Priority: P1)

KISDI 연구원이 시스템을 실행하면, 정책 키워드(플랫폼 규제, 공정거래, 개인정보, 플랫폼 노동, 저작권, AI 규제 등) 기반으로 Naver Search API를 통해 최신 뉴스 기사가 자동 수집된다. 수집된 기사에는 플랫폼명(네이버, 카카오, 쿠팡 등)과 기관명 태깅이 자동으로 수행된다.

**Why this priority**: 보도자료와 함께 뉴스 기사는 분석의 핵심 데이터 소스이다.

**Independent Test**: 정책 키워드로 검색한 뉴스 기사가 JSON 파일로 저장되고, 각 항목에 제목·요약·원문링크·날짜·플랫폼태그·기관태그가 포함되어 있는지 확인한다.

**Acceptance Scenarios**:

1. **Given** Naver API 인증 정보가 설정된 상태, **When** 뉴스 수집 명령을 실행하면, **Then** 정책 키워드별로 최신순 뉴스가 수집되어 JSON 파일로 저장된다
2. **Given** 수집된 뉴스 기사 목록, **When** 후처리가 완료되면, **Then** 각 기사에 해당 플랫폼명과 기관명이 태그로 부착된다
3. **Given** 동일한 기사가 다른 키워드 검색에서 중복 수집된 경우, **When** 저장 처리가 완료되면, **Then** 중복 기사는 하나만 저장되고 관련 키워드 태그는 병합된다

---

### User Story 3 - LLM 기반 뉴스 분석 (Priority: P2)

KISDI 연구원이 수집된 뉴스·보도자료에 대해 분석 명령을 실행하면, Google Gemini API를 활용하여 각 기사의 요약, 핵심 키워드 추출, 감성 분석(긍정/부정/중립), 7대 정책영역 멀티라벨 분류가 자동으로 수행된다. 분석 결과에는 신뢰도 점수가 포함되며, 원문에 없는 내용이 생성되지 않도록 한다.

**Why this priority**: 수집된 데이터를 정책적으로 활용 가능한 형태로 변환하는 핵심 단계이다.

**Independent Test**: 수집된 기사 10건에 대해 분석을 실행하고, 각 기사에 요약·키워드·감성·정책태그·신뢰도가 부여되었는지, 원문 링크가 보존되었는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 수집된 뉴스·보도자료 데이터, **When** 분석 명령을 실행하면, **Then** 각 항목에 한국어 요약(3문장 이내), 핵심 키워드(5개 이내), 감성 레이블, 정책영역 태그가 생성된다
2. **Given** 하나의 기사가 복수의 정책영역에 해당하는 경우, **When** 정책 분류가 수행되면, **Then** 해당하는 모든 영역이 멀티라벨로 태깅된다
3. **Given** 분석 완료된 결과, **When** 결과를 확인하면, **Then** 각 분석 항목에 신뢰도 점수(0~1)가 포함되고, 원문 링크가 보존되어 있다

---

### User Story 4 - 리스크 스코어링 및 급상승 이슈 탐지 (Priority: P3)

KISDI 연구원이 분석된 데이터를 기반으로 리스크 평가를 실행하면, 각 이슈별 리스크 점수가 산출되고, 급상승 이슈(언급량 급증, 규제기관 동반 언급, 감성 급변)가 자동으로 탐지된다. 리스크 점수가 임계값 이상인 이슈는 "사람 확인 필요"로 표시된다.

**Why this priority**: 정책 포럼에서 즉각 대응이 필요한 이슈를 식별하기 위한 핵심 기능이다.

**Independent Test**: 분석된 데이터 세트에 대해 리스크 평가를 실행하고, 각 이슈에 리스크 점수가 부여되고, 임계값 이상 이슈에 "사람 확인 필요" 플래그가 설정되었는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 분석 완료된 뉴스·보도자료 데이터, **When** 리스크 평가를 실행하면, **Then** 각 이슈 클러스터에 리스크 점수(0~100)가 산출된다
2. **Given** 특정 이슈의 언급량이 전주 대비 200% 이상 증가한 경우, **When** 급상승 탐지가 수행되면, **Then** 해당 이슈가 급상승 이슈로 표시된다
3. **Given** 리스크 점수가 임계값(기본값 70) 이상인 이슈, **When** 결과가 출력되면, **Then** "사람 확인 필요" 플래그가 함께 표시된다

---

### User Story 5 - 주간 브리핑 및 리스크 히트맵 생성 (Priority: P3)

KISDI 연구원이 리포트 생성 명령을 실행하면, TOP 10 이슈 브리핑(이슈 요약, 근거 링크, 정책 질문 자동 도출)과 리스크 히트맵(플랫폼 x 정책영역 매트릭스)이 자동으로 생성된다. 리포트는 포럼 위원 및 연구진이 바로 활용할 수 있는 형태이다.

**Why this priority**: 최종 산출물로, 수집·분석·평가 단계가 완료된 후에 가능하다.

**Independent Test**: 분석·평가 완료된 데이터에 대해 리포트를 생성하고, TOP 10 이슈 목록과 히트맵이 포함된 파일이 생성되었는지, 각 이슈에 원문 링크가 첨부되었는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 리스크 평가가 완료된 데이터, **When** 브리핑 생성을 실행하면, **Then** TOP 10 이슈 요약이 생성되고 각 이슈에 근거 원문 링크가 포함된다
2. **Given** 브리핑 생성이 완료되면, **When** 리포트를 확인하면, **Then** 각 이슈에 대해 정책 관련 질문이 1~2개 자동 도출되어 있다
3. **Given** 분석된 데이터에 플랫폼명과 정책영역 태그가 있는 경우, **When** 히트맵 생성을 실행하면, **Then** 플랫폼 x 정책영역 매트릭스 형태의 리스크 히트맵이 생성된다

---

### Edge Cases

- RSS 피드가 일시적으로 접속 불가능한 경우 해당 기관을 건너뛰고 나머지를 수집한다
- Naver API 일일 호출 한도(25,000회)에 도달한 경우 경고 메시지를 표시하고 수집을 중단한다
- Gemini API 호출 실패(네트워크 오류, 할당량 초과) 시 해당 항목을 "미분석"으로 표시하고 재시도 가능하도록 한다
- 수집된 기사 본문이 비어 있거나 너무 짧은 경우(100자 미만) 분석 대상에서 제외한다
- hwpx 파일 파싱 실패 시 해당 항목을 "첨부파일 파싱 실패"로 표시하고 RSS 본문만 사용한다
- 최초 실행 시 이전 주 데이터가 없는 경우, 급상승 이슈 탐지를 건너뛰고 리스크 점수만 산출한다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 정책브리핑(korea.kr) RSS 피드를 통해 7개 대표 기관의 보도자료를 자동 수집해야 한다
- **FR-002**: 시스템은 수집된 보도자료의 hwpx 첨부파일이 있는 경우 텍스트를 추출하여 본문에 포함해야 한다
- **FR-003**: 시스템은 Naver Search API를 통해 정책 키워드 기반으로 뉴스 기사를 수집해야 하며, 키워드 목록은 설정 파일(JSON/YAML)로 관리하여 코드 수정 없이 변경 가능해야 한다
- **FR-004**: 시스템은 수집된 뉴스 기사에 플랫폼명(국내·해외)과 기관명을 자동 태깅해야 한다
- **FR-005**: 시스템은 동일 URL을 기준으로 중복 기사를 탐지하고 제거해야 한다 (다른 매체의 유사 보도는 클러스터링 단계에서 그룹화)
- **FR-006**: 시스템은 Google Gemini API를 활용하여 각 기사의 요약, 키워드 추출, 감성 분석, 정책영역 멀티라벨 분류를 수행해야 한다
- **FR-007**: 시스템은 각 분석 결과에 신뢰도 점수를 포함해야 한다
- **FR-008**: 시스템은 이슈별 리스크 점수를 산출하고, 급상승 이슈를 자동 탐지해야 한다
- **FR-009**: 시스템은 리스크 점수가 임계값 이상인 이슈를 "사람 확인 필요"로 표시해야 한다
- **FR-010**: 시스템은 TOP 10 이슈 브리핑을 Markdown 파일로 자동 생성하고, 리스크 히트맵(플랫폼 x 정책영역)은 Markdown 테이블 + PNG 이미지로 함께 생성해야 한다
- **FR-011**: 시스템의 모든 요약·리포트에는 반드시 원문 링크가 첨부되어야 한다
- **FR-012**: 시스템은 모든 수집 데이터에 수집 시각, 출처 URL, 원문 제목을 메타데이터로 보존해야 한다
- **FR-013**: 시스템은 각 파이프라인 단계(수집, 전처리, 분석, 평가, 리포트)를 독립적으로 실행할 수 있어야 한다
- **FR-014**: 시스템은 API 인증 정보를 환경 변수 또는 설정 파일로 관리해야 한다
- **FR-015**: 시스템은 LLM API 비용 절감을 위해 경량 모델(gemini-2.5-flash-lite)을 기본으로 사용해야 한다

### Key Entities

- **Article (기사/보도자료)**: 수집된 개별 뉴스 기사 또는 보도자료. 제목, 본문, 출처 URL, 게시일, 수집일, 출처 유형(RSS/API), 기관명 포함
- **Analysis (분석 결과)**: 기사에 대한 LLM 분석 결과. 요약, 키워드 목록, 감성 레이블, 정책영역 태그(멀티라벨), 신뢰도 점수 포함. Article과 1:1 관계
- **Issue Cluster (이슈 클러스터)**: LLM 추출 키워드의 일치/유사도를 기반으로 유사 주제의 기사들을 묶은 그룹. 클러스터 대표 키워드, 소속 기사 목록, 리스크 점수, 급상승 여부 포함
- **Briefing Report (브리핑 리포트)**: 주간 단위 자동 생성 리포트. TOP 10 이슈 요약, 정책 질문, 리스크 히트맵 데이터 포함. 생성일 기준

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 7개 기관의 보도자료가 수집 명령 1회 실행으로 모두 수집된다
- **SC-002**: 정책 키워드 기반 뉴스 수집 시 키워드당 최대 1,000건까지 최신순으로 수집 가능하다
- **SC-003**: 중복 기사 제거율 95% 이상 (동일 기사가 다른 키워드에서 중복 수집된 경우)
- **SC-004**: 수집된 기사의 100%에 원문 링크가 보존된다
- **SC-005**: LLM 분석 결과의 정책영역 분류 정확도가 수동 검증 대비 80% 이상이다
- **SC-006**: 리스크 임계값 이상 이슈의 "사람 확인 필요" 표시 누락률 0%이다
- **SC-007**: 브리핑 리포트의 모든 이슈 요약에 근거 원문 링크가 포함된다
- **SC-008**: 시스템 전체 파이프라인(수집~리포트)이 Windows 10/11 랩탑에서 정상 실행된다

## Clarifications

### Session 2026-03-31

- Q: 기사 중복 판별 기준은? → A: URL 기반 — 동일 URL만 중복으로 판단하고, 매체 간 유사 보도는 클러스터링 단계에서 그룹화
- Q: 브리핑 리포트 출력 형식은? → A: Markdown 파일 기반, 리스크 히트맵은 Markdown 테이블 + PNG 이미지 병행
- Q: 이슈 클러스터링 방법은? → A: LLM 추출 키워드의 일치/유사도 기반 그룹화
- Q: 검색 키워드 목록 관리 방식은? → A: 설정 파일(JSON/YAML)로 관리, 코드 수정 없이 변경 가능

## Assumptions

- KISDI 연구원의 Windows 랩탑에 Python 3.10 이상이 설치되어 있다
- 인터넷 접속이 가능하며, korea.kr RSS 피드와 Naver API, Google Gemini API에 접근 가능하다
- Naver 개발자 센터에서 Client ID/Secret을 발급받아 사용한다 (https://developers.naver.com 에서 애플리케이션 등록 후 "검색" API 사용 설정)
- Google Gemini API 키를 Google AI Studio에서 발급받아 사용한다 (무료 티어 또는 유료 플랜)
- Gemini API 비용 절감을 위해 gemini-2.5-flash-lite 모델을 기본으로 사용한다
- 데이터는 로컬 파일 시스템(JSON)에 저장하며, 별도 데이터베이스 서버는 사용하지 않는다
- RSS 피드는 기관당 최신 50건을 제공하므로, 주기적 수집(일 1회 이상)으로 누락을 최소화한다
- hwpx 파일 텍스트 추출은 Python 라이브러리를 활용하되, 파싱 실패 시 RSS 본문으로 대체한다
- 시스템은 CLI 기반으로 동작하며, 웹 UI는 범위 밖이다
